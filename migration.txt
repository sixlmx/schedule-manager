# Шаблон SQL-миграции для schedule-manager-vanilla

## Назначение

Этот файл — учебно-практический шаблон для будущих PostgreSQL-миграций
и стартовых справочных данных.

Он нужен, чтобы заранее зафиксировать аккуратный формат изменений базы:
– что меняется;
– какие таблицы затрагиваются;
– какие данные добавляются;
– как проверить изменение перед применением.

## Важно

– Файл не подключён к приложению автоматически.
– Файл не применяется к базе сам по себе.
– Перед применением любой миграции её должен проверить разработчик.
– Этот файл не заменяет исходный SQL-дамп проекта.
– Этот файл не создаёт полноценную систему миграций, а задаёт понятный шаблон.

## Пример структуры миграции

-- 001_seed_reference_data.sql

BEGIN;

-- Преподаватели
INSERT INTO public.teachers (name, fio, "position", color)
VALUES
  ('Иванов Иван Иванович', 'Иванов И. И.', 'преподаватель', NULL),
  ('Петров Пётр Петрович', 'Петров П. П.', 'старший преподаватель', NULL);

-- Предметы
INSERT INTO public.subjects (name, abbreviation)
VALUES
  ('Математика', 'МАТ'),
  ('Информатика', 'ИНФ');

-- Группы
INSERT INTO public.groups (name, abbreviation, year_of_admission)
VALUES
  ('ИС-101', 'ИС-101', 2026),
  ('ПКС-201', 'ПКС-201', 2025);

-- Звонки
INSERT INTO public.bells (lesson_number, start_time, end_time)
VALUES
  (1, '08:30:00', '09:15:00'),
  (2, '09:25:00', '10:10:00'),
  (3, '10:20:00', '11:05:00'),
  (4, '11:15:00', '12:00:00'),
  (5, '12:10:00', '12:55:00'),
  (6, '13:05:00', '13:50:00'),
  (7, '14:00:00', '14:45:00');

COMMIT;

## Пояснения к SQL

– BEGIN открывает транзакцию: изменения либо применятся все вместе, либо не применятся.
– COMMIT завершает транзакцию и сохраняет изменения.
– INSERT INTO добавляет строки в указанную таблицу.
– VALUES перечисляет данные, которые нужно вставить.
– NULL означает отсутствие значения.
– public указывает схему базы данных.
– "position" написано в двойных кавычках, потому что это точное имя столбца из текущей схемы.

## Правила для будущих миграций

– Одна миграция должна решать одну понятную задачу.
– Название миграции должно отражать её смысл и порядок выполнения.
– Изменения схемы, seed-данные, frontend и backend не нужно смешивать в одном PR.
– Перед применением миграции нужно проверить её на тестовой базе.
– После подготовки миграции нужно проверить git diff.
– Перед отправкой PR нужно убедиться, что в коммит попали только файлы текущей задачи.
